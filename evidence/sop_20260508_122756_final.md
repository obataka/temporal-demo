#### 4.2. リトライ機構とフェーズ遷移
`attempt`カウントを用いたリトライ時の処理（`_retry`サフィックスの利用）と、フェーズ間の論理的な遷移について説明します。
各フェーズは、人間からのフィードバックに基づいて再試行される可能性があります。本Activityは、入力パラメータ `request.attempt` を利用してこのリトライ状況を識別します。

*   **`attempt` カウント:** `request.attempt` は、現在の試行回数を表します。
    *   `attempt = 0`: 初回の実行。
    *   `attempt > 0`: リトライ実行。
*   **`_retry` サフィックスの利用:** `_SYSTEM` 辞書（後述の6.1節で詳細）では、`"outline"`, `"draft"`, `"review"` に加えて、それぞれに対応するリトライ用のキー (`"outline_retry"`, `"draft_retry"`, `"review_retry"`) が定義されています。
    *   `request.attempt > 0` の場合、`f"{request.phase}_retry"` というキーが使用され、LLMに「前回のアウトライン/草稿に対してフィードバックがあるので、それを反映して改善してください」という指示が与えられます。
*   **フェーズ間の論理的な遷移:** ワークフローレベルでは、`outline` → `draft` → `review` という順序でフェーズが進行します。各フェーズ内でのリトライは、同じフェーズキー (`outline`, `draft`, `review`) を使いつつ、`attempt` カウントを増やして本Activityを再呼び出しすることで実現されます。これにより、LLMは前回の出力とフィードバックを参照して、より良い出力を生成しようと試みます。

#### 4.3. LLMに与える役割の切り替え
`_SYSTEM`辞書によるLLMの「役割」定義が、どのようにフェーズとリトライ状況に応じて切り替わるかを解説します。
`_SYSTEM` 辞書は、各フェーズおよびリトライ状況に応じてLLMに与える「役割（System Instruction）」を定義しています。

```python
_SYSTEM = {
    "outline": "あなたは経験豊富なドキュメントアーキテクトです。...",
    "outline_retry": "あなたは経験豊富なドキュメントアーキテクトです。前回のアウトラインに対して人間からフィードバックが届いています。...",
    "draft": "あなたは熟練した技術ライターです。...",
    "draft_retry": "あなたは熟練した技術ライターです。前回のSOP草稿に対して人間からフィードバックが届いています。...",
    "review": "あなたは品質保証の専門家です。...",
    "review_retry": "あなたは品質保証の専門家です。前回のレビュー結果に対して人間からフィードバックが届いています。...",

```
*   **フェーズごとの役割:**
    *   `outline` フェーズでは「ドキュメントアーキテクト」として構造設計に集中させます。
    *   `draft` フェーズでは「技術ライター」として詳細なコンテンツ作成に集中させます。
    *   `review` フェーズでは「品質保証の専門家」として批判的視点での評価に集中させます。
*   **リトライ時の役割強化:** リトライ時 (`_retry` サフィックス付きのキー) の指示には、「人間からのフィードバックを最優先で反映する」という強調が加えられ、LLMが単に再生成するのではなく、具体的な指示に従って改善を行うよう促しています。

この動的な役割の切り替えにより、各フェーズでLLMの能力を最大限に引き出し、高品質なSOP生成に貢献します。

#### 5.2. 入力データモデル `SOPRequest`
`SOPRequest`クラスの構造（`topic`, `phase`, `source_code`, `outline`, `draft`, `previous_output`, `feedback`, `attempt`など）とその各フィールドの役割を解説します。
`SOPRequest` クラスは、`pydantic.BaseModel` を継承しており、Activityへの入力データを構造化してバリデーションします。

```python
# core/models.py (SOPRequestの定義を想定)
from pydantic import BaseModel, Field
from typing import Optional

class SOPRequest(BaseModel):
    topic: str = Field(..., description="SOPが対象とするトピックまたは機能の簡潔な説明。")
    phase: str = Field(..., description="現在のSOP生成フェーズ ('outline', 'draft', 'review')。")
    source_code: Optional[str] = Field(None, description="ドキュメント化対象のソースコード。")
    outline: Optional[str] = Field(None, description="承認済みのアウトライン（draft/reviewフェーズで使用）。")
    draft: Optional[str] = Field(None, description="SOPの草稿（reviewフェーズで使用）。")
    previous_output: Optional[str] = Field(None, description="前回のLLM出力（リトライ時に使用）。")
    feedback: Optional[str] = Field(None, description="人間からのフィードバック（リトライ時に使用）。")
    attempt: int = Field(0, description="現在の試行回数（初回は0）。")
```

*   **`topic` (str):** SOPが対象とするシステムや機能の簡潔な説明。LLMにSOPの主題を伝えるために使用されます。
*   **`phase` (str):** 現在のSOP生成フェーズ。`"outline"`, `"draft"`, `"review"` のいずれかです。この値に基づいてLLMのシステム指示とプロンプトが構築されます。
*   **`source_code` (Optional[str]):** ドキュメント化の対象となるソースコードの文字列。`outline` および `draft` フェーズでLLMに提供されます。
*   **`outline` (Optional[str]):** 以前の `outline` フェーズで生成され、承認されたアウトライン。`draft` フェーズでSOP本文の構造の指針としてLLMに提供されます。
*   **`draft` (Optional[str]):** 以前の `draft` フェーズで生成されたSOPの草稿。`review` フェーズでLLMによるレビュー対象となります。
*   **`previous_output` (Optional[str]):** 前回のActivity実行でLLMから生成された出力。リトライ時 (`attempt > 0`) に、LLMが前回の結果を参照し、フィードバックを適用する際に使用されます。
*   **`feedback` (Optional[str]):** 人間（または自動化された評価システム）からのフィードバックテキスト。リトライ時 (`attempt > 0`) に、LLMが改善を行うための具体的な指示として使用されます。
*   **`attempt` (int):** 現在のActivityの試行回数。初回呼び出しは `0` で、リトライするごとにインクリメントされます。この値は、`_SYSTEM` 辞書から適切なシステム指示を選択するために使用されます。

これらのフィールドは、LLMに与える情報の粒度と関連性を制御し、各フェーズでのLLMの応答を最適化するために設計されています。

#### 5.3. 出力データモデル `LLMResult`
`LLMResult`クラスの構造（`text`, `model`, `input_tokens`, `output_tokens`, `total_tokens`, `latency_ms`など）と、LLMからの応答情報をどのように保持するかを説明します。
`LLMResult` クラスは、Gemini APIからの応答と、その呼び出しに関するメトリクスを構造化して格納します。

```python
# core/models.py (LLMResultの定義を想定)
from pydantic import BaseModel, Field
from typing import Optional

class LLMResult(BaseModel):
    text: str = Field(..., description="LLMからの生成テキスト。")
    model: str = Field(..., description="使用されたLLMモデルの識別子。")
    input_tokens: int = Field(..., description="LLMへの入力に消費されたトークン数。")
    output_tokens: int = Field(..., description="LLMの出力に消費されたトークン数。")
    total_tokens: int = Field(..., description="合計トークン数 (入力 + 出力)。")
    latency_ms: float = Field(..., description="LLM API呼び出しのレイテンシ（ミリ秒）。")
    error: Optional[str] = Field(None, description="エラーが発生した場合のエラーメッセージ。")
```

*   **`text` (str):** Geminiモデルが生成した主なテキスト応答。これはアウトライン、SOP草稿、またはレビュー結果となります。
*   **`model` (str):** 応答を生成するために使用されたGeminiモデルのID（例: `"gemini-2.5-flash"`）。
*   **`input_tokens` (int):** プロンプト（入力）に費やされたトークン数。Gemini APIの `usage_metadata.prompt_token_count` から取得されます。
*   **`output_tokens` (int):** 生成された応答（出力）に費やされたトークン数。Gemini APIの `usage_metadata.candidates_token_count` から取得されます。
*   **`total_tokens` (int):** `input_tokens` と `output_tokens` の合計。APIコストの計算に直接関係します。Gemini APIの `usage_metadata.total_token_count` から取得されます。
*   **`latency_ms` (float):** Gemini APIへのリクエスト送信から応答受信までの時間（ミリ秒単位）。パフォーマンス監視に有用です。
*   **`error` (Optional[str]):** エラーが発生した場合に、エラーメッセージを格納します。成功時には `None` となります。

この構造化された出力により、WorkflowはLLMの応答を容易にパースして後続の処理に利用できるだけでなく、コストやパフォーマンスに関する重要なメトリクスも収集できます。

### 6. コア機能の実装
*   **対象読者:** 開発者
*   **所要時間の目安 (本セクション):** 30分
本Activityの中核をなす、システム指示の管理、LLMコンテンツの動的構築、およびGemini API呼び出しの実装について深掘りします。

#### 6.1. システム指示（System Instructions）の管理 (`_SYSTEM`辞書)
`_SYSTEM` 辞書は、各SOP生成フェーズおよびリトライの有無に応じて、Geminiモデルに与える「システム指示」を定義します。

```python
_SYSTEM = {
    "outline": (
        "あなたは経験豊富なドキュメントアーキテクトです。"
        "与えられたソースコードを分析し、標準作業手順書（SOP）の章立て（アウトライン）を提案してください。"
        "各章のタイトルと1〜2行の説明を含め、明確で構造化されたアウトラインをMarkdownで出力してください。"
    ),
    "outline_retry": (
        "あなたは経験豊富なドキュメントアーキテクトです。"
        "前回のアウトラインに対して人間からフィードバックが届いています。"
        "そのフィードバックを最優先で反映し、改善されたアウトラインをMarkdownで提供してください。"
    ),
    # ... 他のフェーズの定義 ...
}
```

*   **設計思想:**
    *   **役割ベースのプロンプティング:** LLMに特定の専門家（ドキュメントアーキテクト、技術ライター、品質保証専門家）の役割を与えることで、その専門性に応じた思考プロセスと出力形式を促します。
    *   **具体的な指示:** 各フェーズでLLMに何を期待するか（例: 「章立てを提案」「詳細を執筆」「レビューし改善点と最終版を提供」）を明確に記述します。
    *   **出力形式の指定:** 「Markdownで出力してください」と明示することで、構造化された応答を期待します。
    *   **リトライ時の指示強化:** リトライ (`_retry` サフィックス) 時には「フィードバックを最優先で反映する」という文言を加え、改善に対するLLMの集中を促します。
*   **`system_instruction` の選択ロジック:**
    `generate_sop_phase_activity` 関数内で、`request.phase` と `request.attempt` に基づいて適切な `system_instruction` が選択されます。

    ```python
    phase_key = f"{request.phase}_retry" if request.attempt > 0 else request.phase
    system_instruction = _SYSTEM[phase_key]
    ```
    これにより、初回実行時は `request.phase` に対応するシステム指示が、リトライ時は `request.phase` に `_retry` を付加したシステム指示が選択されます。

この管理されたシステム指示により、LLMは各フェーズの目的に合致した、高品質で一貫性のある応答を生成することが可能になります。

#### 6.2. LLMプロンプトコンテンツの動的生成 (`_build_contents`関数)
`_build_contents` 関数は、`SOPRequest` オブジェクトからGeminiモデルに渡すための主要なプロンプトコンテンツを動的に構築します。

```python
def _build_contents(request: SOPRequest) -> str:
    phase = request.phase
    is_retry = request.attempt > 0

    if phase == "outline":
        base = (
            f"## ドキュメント化対象\n{request.topic}\n\n"
            f"## ソースコード\n```python\n{request.source_code}\n```"
        )
        if is_retry:
            return (
                f"{base}\n\n"
                f"## 前回のアウトライン（試行 #{request.attempt - 1}）\n"
                f"{request.previous_output}\n\n"
                f"## 人間からのフィードバック\n{request.feedback}\n\n"
                "---\nフィードバックを反映した改善版アウトラインを提供してください。"
            )
        return base + "\n\n---\n上記ソースコードの SOP アウトラインを提案してください。"

    elif phase == "draft":
        # ... draftフェーズのロジック ...

    else:  # review
        # ... reviewフェーズのロジック ...
```

*   **関数の役割:** `_build_contents` は、`SOPRequest` の各フィールド（`topic`, `source_code`, `outline`, `draft`, `previous_output`, `feedback`）を組み合わせ、現在のフェーズとリトライ状況に応じた最適なプロンプトテキストを生成します。
*   **フェーズごとのプロンプト構造:**
    *   **`outline` フェーズ:**
        *   初回 (`is_retry` が `False`): ドキュメント化対象のトピックとソースコードをLLMに提示し、「SOPアウトラインを提案してください」と指示します。
        *   リトライ (`is_retry` が `True`): 上記に加え、前回のLLM出力 (`request.previous_output`) と人間からのフィードバック (`request.feedback`) を提示し、「フィードバックを反映した改善版アウトラインを提供してください」と明確に指示します。
    *   **`draft` フェーズ:**
        *   初回: ドキュメント化対象、承認済みアウトライン (`request.outline`)、およびソースコードを提示し、「詳細なSOP本文を執筆してください」と指示します。
        *   リトライ: 上記に加え、前回の草稿とフィードバックを提示し、「フィードバックを反映した改善版草稿を提供してください」と指示します。
    *   **`review` フェーズ:**
        *   初回: ドキュメント化対象とSOP草稿 (`request.draft`) を提示し、「このSOP草稿をレビューし、【改善点リスト】と【最終版SOP】を提供してください」と指示します。
        *   リトライ: 上記に加え、前回のレビュー結果とフィードバックを提示し、「フィードバックを反映した改善された最終版を提供してください」と指示します。
*   **コード例（`draft` フェーズ）:**

    ```python
    elif phase == "draft":
        base = (
            f"## ドキュメント化対象\n{request.topic}\n\n"
            f"## 承認済みアウトライン\n{request.outline}\n\n"
            f"## ソースコード\n```python\n{request.source_code}\n```"
        )
        if is_retry:
            return (
                f"{base}\n\n"
                f"## 前回の草稿（試行 #{request.attempt - 1}）\n"
                f"## {request.previous_output}\n\n"
                f"## 人間からのフィードバック\n{request.feedback}\n\n"
                "---\nフィードバックを反映した改善版草稿を提供してください。"
            )
        return base + "\n\n---\n上記アウトラインに基づき、詳細なSOP本文を執筆してください。"
    ```
    この例から分かるように、各フェーズで必要な入力情報が異なるため、`_build_contents` 関数は非常に柔軟にプロンプトを構築します。Markdownのヘッダー (`##`) を使用して情報を構造化し、LLMが各セクションの内容を正確に理解できるように工夫されています。

#### 6.3. Gemini API呼び出しと応答処理
`google.genai`クライアントの初期化、モデル指定（`_MODEL`）、`system_instruction`の適用、`client.models.generate_content`の呼び出し、および応答から`LLMResult`を構築するプロセスを説明します。
`generate_sop_phase_activity` 関数内で、`google.generativeai` クライアントを使用してGemini APIを呼び出し、その応答を処理します。

```python
async def generate_sop_phase_activity(request: SOPRequest) -> LLMResult:
    from google import genai
    from google.genai import types

    api_key = os.environ.get("GEMINI_API_KEY")
    if not api_key:
        raise EnvironmentError("GEMINI_API_KEY が設定されていません。")

    # genai.Client の初期化自体は同期的に行われますが、
    # その後の generate_content メソッドは非同期です。
    client = genai.Client(api_key=api_key)

    phase_key = f"{request.phase}_retry" if request.attempt > 0 else request.phase
    system_instruction = _SYSTEM[phase_key]
    contents = _build_contents(request)

    start = time.monotonic()
    # 非同期で generate_content を呼び出すため、await を使用します
    response = await client.models.generate_content(
        model=_MODEL,
        contents=contents,
        config=types.GenerateContentConfig(system_instruction=system_instruction),
    )
    latency_ms = (time.monotonic() - start) * 1000

    usage = response.usage_metadata
    return LLMResult(
        text=response.text,
        model=_MODEL,
        input_tokens=getattr(usage, "prompt_token_count", 0) or 0,
        output_tokens=getattr(usage, "candidates_token_count", 0) or 0,
        total_tokens=getattr(usage, "total_token_count", 0) or 0,
        latency_ms=round(latency_ms, 2),
    )
```

*   **`google.genai` クライアントの初期化:**
    ```python
    api_key = os.environ.get("GEMINI_API_KEY")
    if not api_key:
        raise EnvironmentError("GEMINI_API_KEY が設定されていません。")
    client = genai.Client(api_key=api_key)
    ```
    環境変数 `GEMINI_API_KEY` からAPIキーを取得し、`genai.Client` を初期化します。キーが設定されていない場合は `EnvironmentError` を発生させます。`genai.Client` 自体の初期化は同期的に行われますが、その後の `client.models.generate_content` メソッドは非同期で実行されます。
*   **モデル指定 (`_MODEL`):**
    ```python
    _MODEL = "gemini-2.5-flash"
    # ...
    response = await client.models.generate_content(
        model=_MODEL,
        # ...
    )
    ```
    事前に定義された `_MODEL` 変数 (`"gemini-2.5-flash"`) を使用して、呼び出すGeminiモデルを指定します。このモデルは高速かつコスト効率の良いテキスト生成に適しています。
*   **`system_instruction` の適用:**
    ```python
    # ...
    config=types.GenerateContentConfig(system_instruction=system_instruction),
    ```
    LLMに役割と振る舞いを指示する `system_instruction` は、`GenerateContentConfig` オブジェクトを通じてAPIリクエストに含められます。これはLLMの応答の質を大きく左右する重要な要素です。
*   **`client.models.generate_content` の呼び出し:**
    ```python
    response = await client.models.generate_content( # await を使用
        model=_MODEL,
        contents=contents,
        config=types.GenerateContentConfig(system_instruction=system_instruction),
    )
    ```
    `client.models.generate_content` メソッドを非同期で呼び出します。
    *   `model`: 使用するGeminiモデル。
    *   `contents`: `_build_contents` 関数で生成された主要なプロンプトテキスト。
    *   `config`: `system_instruction` を含む追加設定。
*   **応答処理と `LLMResult` の構築:**
    ```python
    latency_ms = (time.monotonic() - start) * 1000
    usage = response.usage_metadata
    return LLMResult(
        text=response.text,
        model=_MODEL,
        input_tokens=getattr(usage, "prompt_token_count", 0) or 0,
        output_tokens=getattr(usage, "candidates_token_count", 0) or 0,
        total_tokens=getattr(usage, "total_token_count", 0) or 0,
        latency_ms=round(latency_ms, 2),
    )
    ```
    API応答から以下の情報を抽出します。
    *   `response.text`: LLMが生成したテキスト。
    *   `response.usage_metadata`: トークン数に関する情報（`prompt_token_count`, `candidates_token_count`, `total_token_count`）。`getattr()` と `or 0` を使用して、これらの属性が存在しない場合でも安全にデフォルト値（0）を取得します。
    *   `time.monotonic()` を使用してAPI呼び出しの開始と終了時刻を記録し、レイテンシを計算します。
    これらの情報を用いて `LLMResult` オブジェクトを構築し、Activityの戻り値として返します。

このセクションで説明されたコア機能の実装は、本ActivityがSOP生成プロセスの各フェーズでLLMを効果的に利用するための基盤を形成します。

### 7. エラーハンドリングとリトライ機構の設計
*   **対象読者:** 開発者、運用担当者
*   **所要時間の目安 (本セクション):** 15分
本Activityは、外部サービスであるGemini APIに依存するため、エラーハンドリングとリトライ機構は堅牢な運用に不可欠です。本章では、エラー処理の戦略と、Temporalの組み込みリトライメカニズムをどのように活用し、設計するかについて解説します。

#### 7.1. 環境設定エラーの処理 (`GEMINI_API_KEY`未設定)
Gemini APIを呼び出すには、有効なAPIキーが環境変数 `GEMINI_API_KEY` に設定されている必要があります。

```python
    api_key = os.environ.get("GEMINI_API_KEY")
    if not api_key:
        raise EnvironmentError("GEMINI_API_KEY が設定されていません。")
```

*   **エラー発生条件:** `os.environ.get("GEMINI_API_KEY")` が `None` または空文字列を返す場合、この条件が真となり `EnvironmentError` が発生します。
*   **設計上の考慮事項:**
    *   この種のエラーは、環境の初期設定ミスに起因し、Activityを再試行しても解決することはありません。そのため、このようなエラーが発生した場合は、速やかにWorkflowを失敗させるべきです。
    *   Temporalの `RetryPolicy` において、`EnvironmentError` を `non_retryable_error_types` に含めることで、無駄なリソース消費を防ぎ、運用担当者に迅速な介入を促すことができます。
*   **対処法:**
    *   **デプロイ前:** Activityを含むWorkerをデプロイする前に、必ず `GEMINI_API_KEY` 環境変数が適切に設定されていることを確認してください。
    *   **運用時:** CI/CDパイプラインやデプロイメントスクリプトで、この環境変数が安全かつ確実に設定されるように自動化を検討してください。Kubernetes Secretsなどのシークレット管理サービスを利用することで、APIキーの漏洩リスクを低減できます。

#### 7.2. Gemini API呼び出し時の潜在的エラーとその対策
Gemini API呼び出し時には、様々なエラーが発生する可能性があります。これらのエラーはActivityの未処理例外としてWorkflowに伝播し、Temporalのリトライポリシーによって処理されます。

1.  **ネットワークエラー:**
    *   **原因:** APIエンドポイントへの接続問題、DNS解決失敗、タイムアウトなど、一時的なインフラストストラクチャの問題。
    *   **設計と対策:** これらは一時的な問題であるため、Temporalの組み込みリトライポリシーが非常に有効です。`initial_interval` と `backoff_coefficient` を適切に設定することで、自動的に再試行され、多くの場合は回復します。Activity自体に明示的な `try-except` を設けるよりも、Temporalの堅牢なメカニズムに任せるのが一般的です。
2.  **APIレート制限 (Rate Limit):**
    *   **原因:** 短期間にAPIリクエストが集中し、Googleの許容リクエストレートを超過した場合。
    *   **設計と対策:** `google-generativeai` ライブラリは通常、レート制限エラーに対して特定の例外（例: `google.api_core.exceptions.ResourceExhausted`）を発生させます。Temporalの指数バックオフを含むリトライポリシーは、このような場合にリクエスト間隔を徐々に長くしてレート制限を回避するのに役立ちます。Workerプロセス自体にクライアントサイドのレートリミッターを実装し、APIリクエストを制御することも検討できます。
3.  **LLMによる応答生成失敗:**
    *   **原因:** プロンプトが複雑すぎる、LLMが安全ポリシーに違反するコンテンツを検知した、内部エラーなどにより、LLMが有効な応答を生成できなかった場合。
    *   **設計と対策:**
        *   **リトライ:** 一時的なLLM内部の問題であれば、リトライによって解決する可能性があります。
        *   **プロンプトの最適化:** 継続的にこの問題が発生する場合、プロンプトの品質そのものが原因である可能性があります。プロンプトをより明確にし、曖昧さを排除し、LLMにとって理解しやすい形式にすることで、生成失敗のリスクを減らせます。
        *   **フォールバック/人間による介入:** 特定の回数リトライしても成功しない場合、Activityは最終的に失敗します。Workflowは、この失敗を検知して人間による介入を要求するか、別の処理フロー（例: デフォルトコンテンツの生成）に切り替えることを検討すべきです。

#### 7.3. Temporal Activityリトライポリシーの設計と適用
Temporalの組み込みリトライメカニズムは、本Activityの堅牢性を大幅に向上させます。WorkflowからActivityを呼び出す際に、`start_to_close_timeout` と `retry_policy` を適切に設定することが、堅牢なエラーハンドリング設計の中核となります。

**WorkflowからのActivity呼び出しと`RetryPolicy`の設定例:**

```python
from temporalio.client import Client, WorkflowHandle
from temporalio.worker import Worker
from temporalio.common import RetryPolicy
from datetime import timedelta # timedelta をインポート

# ... Activity と Workflow の定義 (省略) ...

async def start_sop_generation_workflow(client: Client, topic: str, source_code: str):
    # Workflow実行の開始 (ここでは Workflow 自体の定義は省略)
    # Workflow内部から generate_sop_phase_activity を呼び出す例を示す

    # generate_sop_phase_activity のためのSOPRequestオブジェクト
    outline_request = SOPRequest(
        topic=topic,
        phase="outline",
        source_code=source_code,
        attempt=0, # 初回試行
    )

    # Activity の実行
    try:
        outline_result: LLMResult = await client.execute_activity(
            generate_sop_phase_activity,
            outline_request,
            task_queue="sop-generation-task-queue",
            # Activityの実行時間制限とリトライポリシーの設計
            start_to_close_timeout=timedelta(minutes=5), # Activityが完了するまでの最大時間
            retry_policy=RetryPolicy(
                initial_interval=timedelta(seconds=1),  # 最初の再試行までの待機時間
                backoff_coefficient=2.0,                # 各再試行で待機時間を乗算する係数 (指数バックオフ)
                maximum_interval=timedelta(minutes=1),  # 再試行間の最大待機時間
                maximum_attempts=5,                     # 最大再試行回数。これを超えるとActivityは最終的に失敗
                non_retryable_error_types=["EnvironmentError", "ValueError"], # 特定のエラーはリトライしない
            ),
            # Activity ID は Workflow の決定論的実行を保証するため重要です。
            # ユニークかつ再試行で同じ ID になるように設計することが推奨されます。
            # 例: f"generate-outline-{workflow.uuid4()}" (Workflow内部から呼び出す場合)
            id="generate-outline-for-my-sop-topic-v1",
        )
        print(f"SOP Outline Generated: {outline_result.text[:100]}...")
        return outline_result
    except Exception as e:
        print(f"Activity failed after retries: {e}")
        raise # Workflow にエラーを再スロー

#     # 通常は Worker が起動し、Workflow から Activity が呼び出されます。
#     # このコードは Workflow 内部からの呼び出し方を具体的に示すためのものです。
#     # 実際に実行する際は、Temporal Service と Worker が動作している必要があります。
#     # client = await Client.connect("localhost:7233")
#     # await start_sop_generation_workflow(client, "My Temporal Workflow", "...")
```

*   **`start_to_close_timeout` の設計:**
    *   Activityが開始されてから完了するまでの最大許容時間です。この時間内に完了しない場合、Activityはタイムアウトと見なされ、リトライポリシーが適用されます。
    *   Gemini APIの応答時間、プロンプトの複雑さ、およびモデルの種類によって変動するため、適切な値を設定することが重要です。例えば、短い応答が期待される場合は数分、複雑な生成にはそれ以上の時間を設定します。
*   **`RetryPolicy` の設計要素:**
    *   **`initial_interval` (timedelta):** 最初の再試行までの待機時間。APIの一時的な障害やレート制限からの回復を考慮し、短すぎず長すぎない時間を設定します。
    *   **`backoff_coefficient` (float):** 各再試行で待機時間を乗算する係数。`1.0` ならば固定間隔、`2.0` ならば指数バックオフとなります。外部APIのレート制限回避には指数バックオフ（`> 1.0`）が効果的です。
    *   **`maximum_interval` (timedelta):** 再試行間の最大待機時間。指数バックオフが際限なく広がるのを防ぎ、リソースを過剰に占有しないように上限を設定します。
    *   **`maximum_attempts` (int):** 最大再試行回数。これを超えるとActivityは最終的に失敗としてWorkflowに報告されます。無限ループを防ぎ、最終的な失敗をWorkflowに伝えるための重要なガードレールです。この回数は、一時的なエラーの頻度やビジネス要件に応じて決定します。
    *   **`non_retryable_error_types` (List[str]):** 指定されたエラータイプが発生した場合は、リトライせず直ちにActivityを失敗とします。
        *   `EnvironmentError` (例: `GEMINI_API_KEY` 未設定): 環境設定の根本的な問題であり、リトライしても解決しないため、ここに含めるのが適切です（前述の7.1節を参照）。
        *   `ValueError` や `TypeError`: Activityへの入力データに問題がある場合など、ビジネスロジックやデータ構造の根本的な不整合を示すエラーも、リトライしても無意味なケースが多いため、ここに追加を検討します。これにより、無駄なリソース消費と時間の浪費を避け、迅速なデバッグと修正を促します。
*   **Activity ID:** Activity IDは、WorkflowがActivityの実行を追跡し、リプレイ時に同じActivityが決定論的に実行されることを保証するために重要です。Workflow内部からActivityを呼び出す際には、`id` パラメータをユニークかつ再現可能な方法で設定することが推奨されます（例: Workflow IDとActivityフェーズ名を組み合わせるなど）。

Temporalのリトライ機構により、Activity自体はビジネスロジックに集中でき、一時的な外部サービスの問題に対する回復性はプラットフォームによって提供されます。`RetryPolicy` を適切に設計することは、システムの安定性と運用効率を大きく左右します。

#### 8.1. Temporal WorkerへのActivity登録
`Worker`インスタンスの作成、タスクキューの指定、および`generate_sop_phase_activity`をWorkerに登録するPythonコード例を示します。
Activityは、`temporalio.worker.Worker` インスタンスに登録され、特定のタスクキューをリッスンするWorkerによって実行されます。

```python

from temporalio.worker import Worker
from temporalio.client import Client

from my_workflows import SOPGenerationWorkflow       # Workflowのインポート (参考)

async def run_worker():
    client = await Client.connect("localhost:7233") # Temporal Serviceに接続

    worker = Worker(
        client,
        task_queue="sop-generation-task-queue", # Activityがリッスンするタスクキュー名
        activities=[generate_sop_phase_activity], # 登録するActivityのリスト
        workflows=[SOPGenerationWorkflow],       # 関連するWorkflowも登録 (通常は同じWorkerで処理)
    )
    print("Starting Worker...")
    await worker.run() # Workerを起動し、タスクキューからのタスクを待機

if __name__ == "__main__":
    # GEMINI_API_KEY 環境変数を設定してから実行してください
    # export GEMINI_API_KEY="YOUR_GEMINI_API_KEY_HERE"
    asyncio.run(run_worker())
```

*   **`Client.connect()`:** Temporal Serviceへの接続を確立します。デフォルトでは `localhost:7233` に接続します。
*   **`Worker` インスタンスの作成:**
    *   `client`: 上記で作成したTemporal Client。
    *   `task_queue`: このWorkerが処理するタスクキューの名前。ActivityとWorkflowは同じタスクキューに登録されます。
    *   `activities`: このWorkerが実行できるActivity関数のリスト。`generate_sop_phase_activity` をここに追加します。
    *   `workflows`: このWorkerが実行できるWorkflow関数のリスト（オプションですが、通常は関連するWorkflowも登録します）。
*   **`worker.run()`:** Workerを起動し、指定されたタスクキューからActivityタスクやWorkflowタスクをポーリングして実行します。

#### 8.2. WorkflowからのActivity呼び出し例
Temporal Workflowから`generate_sop_phase_activity`を呼び出す際の非同期実行（`await workflow.execute_activity`）や、入力パラメータの渡し方をコード例で示します。
Workflowから `generate_sop_phase_activity` を呼び出す際は、`temporalio.workflow.execute_activity` を使用します。

```python
from temporalio import workflow
from datetime import timedelta

from my_activities import generate_sop_phase_activity
from core.models import SOPRequest, LLMResult # データモデルのインポート

@workflow.defn
class SOPGenerationWorkflow:
    @workflow.run
    async def run(self, topic: str, source_code: str) -> str:
        # Outline フェーズ
        outline_request = SOPRequest(
            topic=topic,
            phase="outline",
            source_code=source_code,
            attempt=0, # 初回試行
        )
        outline_result: LLMResult = await workflow.execute_activity(
            generate_sop_phase_activity,
            outline_request,
            start_to_close_timeout=timedelta(minutes=5),
            retry_policy=workflow.RetryPolicy(maximum_attempts=3),
            # Activity ID は Workflow の決定論的実行を保証するため重要です。
            # ユニークかつ再試行で同じ ID になるように設計することが推奨されます。
            # 例: `f"generate-outline-{workflow.uuid4()}"`
            id=f"generate-outline-{workflow.uuid4()}",
        )
        generated_outline = outline_result.text

        workflow.logger.info(f"Generated Outline:\n{generated_outline}")

        # ここで人間によるレビューや承認を待つシグナルを待機するロジックを挟むことも可能
        # 例: await workflow.wait_for_external_signal("human_approval_signal")

        # Draft フェーズ
        draft_request = SOPRequest(
            topic=topic,
            phase="draft",
            source_code=source_code,
            outline=generated_outline, # 承認済みアウトラインを渡す
            attempt=0,
        )
        draft_result: LLMResult = await workflow.execute_activity(
            generate_sop_phase_activity,
            draft_request,
            start_to_close_timeout=timedelta(minutes=15),
            retry_policy=workflow.RetryPolicy(maximum_attempts=3),
            id=f"generate-draft-{workflow.uuid4()}",
        )
        generated_draft = draft_result.text

        workflow.logger.info(f"Generated Draft:\n{generated_draft}")

        # ... review フェーズの呼び出しも同様 ...

        return generated_draft # 例として最終ドラフトを返す
```

*   **`await workflow.execute_activity()`:** WorkflowからActivityを非同期で呼び出します。
    *   第一引数: 呼び出すActivity関数 (`generate_sop_phase_activity`)。
    *   第二引数: Activityへの入力データ（`SOPRequest` オブジェクト）。
    *   `start_to_close_timeout`: Activityの実行タイムアウト。LLMの応答時間に応じて適切な値を設定します。
    *   `retry_policy`: Activity固有のリトライポリシー。Workflowレベルでデフォルト設定も可能ですが、Activityごとに調整することも重要です。
    *   `id`: ActivityのユニークなID。Workflowが決定論的であるために重要です。Workflowリプレイ時に同じActivityが同じIDで実行されることを保証します（詳細は7.3節も参照）。`workflow.uuid4()` などを使用してユニークなIDを生成し、フェーズ名などと組み合わせるのが一般的です。

#### 8.3. デプロイメントとスケーリング
コンテナ化（Docker）、Kubernetesなどの環境でのデプロイ方法と、SOP生成の負荷に応じたWorkerのスケーリング戦略について言及します。
SOP生成システムを本番環境にデプロイする際の考慮事項：

*   **コンテナ化 (Docker):** WorkerアプリケーションをDockerコンテナとしてパッケージ化することを強く推奨します。これにより、環境依存性を排除し、どこでも一貫した動作を保証できます。`Dockerfile` にはPython環境と必要なライブラリのインストール手順、およびWorkerの起動コマンドを含めます。
*   **コンテナオーケストレーション (Kubernetes):** DockerコンテナをKubernetesなどのコンテナオーケストレーションプラットフォームにデプロイすることで、Workerアプリケーションのデプロイ、管理、スケーリングが容易になります。
    *   **Pod定義:** Workerコンテナ、環境変数（特に `GEMINI_API_KEY` をKubernetes Secretsとして）、リソース要求/制限を含むPod定義を作成します。
    *   **Deployment:** 複数のWorker Podを管理し、ローリングアップデートなどを可能にするDeploymentリソースを使用します。
    *   **Service:** 必要に応じて、Workerが外部からアクセスする必要がある場合にServiceを定義します（ただし、通常Temporal WorkerはPullモデルで動作するため不要なことが多い）。
*   **スケーリング:**
    *   SOP生成の負荷（Workflow実行数）に応じて、Workerのレプリカ数を増減させることでスケーリングします。
    *   Kubernetesの場合、Horizontal Pod Autoscaler (HPA) を使用して、CPU使用率やカスタムメトリクス（例: タスクキューのバックログサイズ）に基づいてWorker Podを自動的にスケーリングできます。
    *   Gemini APIのレート制限に注意し、WorkerのスケールアップがAPIリクエストレートの許容範囲内であることを確認してください。

#### 9.1. APIキーのセキュアな管理
`GEMINI_API_KEY` は、Google Gemini APIへのフルアクセス権を持つ機密情報です。以下の方法で安全に管理する必要があります。

*   **環境変数:** 開発環境や小規模なデプロイでは、環境変数として設定することが一般的です。ただし、OSのプロセスリストやログファイルから漏洩するリスクがないことを確認してください。
*   **シークレット管理サービス:** 本番環境では、以下のいずれかの専用シークレット管理サービスを使用することを強く推奨します。
    *   **AWS Secrets Manager:** AWS環境で利用。APIキーを暗号化して保存し、IAMロールに基づいてアクセスを許可します。
    *   **GCP Secret Manager:** Google Cloud Platform環境で利用。同様にキーの暗号化とIAMによるアクセス制御を提供します。
    *   **Kubernetes Secrets:** Kubernetesクラスター内でシークレットを管理します。Base64エンコードされるだけで暗号化はされないため、ディスクレベルの暗号化やRBACポリシーと組み合わせて利用します。
*   **原則:**
    *   **コードにハードコードしない:** 絶対にソースコード内にAPIキーを直接書き込まないでください。
    *   **最小権限の原則:** APIキーには、そのActivityが必要とする最小限の権限のみを付与してください。
    *   **ローテーション:** 定期的にAPIキーをローテーションし、万が一漏洩した場合の影響を最小限に抑えてください。

#### 9.2. コストとレートリミットの最適化
Gemini APIの利用には料金が発生し、またAPI呼び出しにはレートリミットが設定されています。

*   **コストの最適化:**
    *   **モデル選択:** `_MODEL = "gemini-2.5-flash"` を使用しているのは、`gemini-1.5-pro` などのより高性能なモデルに比べてコスト効率が良いためです。SOP生成の要件（品質、速度）に応じて、適切なモデルを選択することが重要です。
    *   **プロンプトの最適化:**
        *   **簡潔性:** 不要な冗長な情報をプロンプトから削除し、トークン数を削減します。
        *   **入力のフィルタリング:** 大量のソースコードをそのまま渡すのではなく、関連性の高い部分のみを抽出して渡すことを検討します。
        *   **アウトラインの短縮:** `outline` や `draft` の情報量を必要最低限に抑えることで、入力トークンを削減します。
    *   **キャッシング:** 同じ入力に対して同じSOPコンテンツを生成する場合、LLMの応答をキャッシュして再利用することでAPI呼び出しを減らし、コストを削減できます。
*   **レートリミットへの対応:**
    *   **リトライとバックオフ:** Temporalの `RetryPolicy` による指数バックオフは、レートリミットエラーに対する最初の防衛線です。
    *   **Workerのスケーリング計画:** 大量のSOPを生成する場合、Workerの数を増やして並列処理を行うことになりますが、その結果としてAPIリクエストレートも増加します。Google CloudのモニタリングツールでAPI使用状況を監視し、必要に応じてレートリミットの引き上げをリクエストするか、クライアントサイドでのレート制御を実装してください。

#### 9.3. ログ監視とパフォーマンスチューニング
システムの健全性と効率性を維持するためには、継続的な監視とパフォーマンスチューニングが不可欠です。

*   **Activityの実行ログ:**
    *   Temporal Workerのログは、Activityの開始、完了、および発生したエラーに関する情報を提供します。構造化ログ（JSON形式など）を採用することで、ログ分析ツールでの検索や集計が容易になります。
    *   `workflow.logger.info()` を用いて、Activityの各フェーズの進捗、LLMからの主要な応答、エラーメッセージなどをログに出力します。
*   **LLMの応答時間 (`latency_ms`):**
    *   `LLMResult` に含まれる `latency_ms` は、Gemini API呼び出しのパフォーマンスを直接測定する重要なメトリクスです。
    *   これらのデータを集計し、平均、中央値、99パーセンタイルなどのレイテンシメトリクスを監視することで、API応答のパフォーマンス傾向を把握し、潜在的なボトルネックを特定できます。
*   **トークン消費量 (`input_tokens`, `output_tokens`, `total_tokens`):**
    *   これらのメトリクスは、APIコストに直接影響します。長期的に監視することで、プロンプトの変更やモデルの選択がコストに与える影響を評価できます。
*   **エラー率:**
    *   Gemini API呼び出しの成功率とエラー率を監視します。エラー率の急激な上昇は、API側の問題、レート制限の超過、またはプロンプトの問題を示唆する可能性があります。
*   **パフォーマンスチューニング:**
    *   **プロンプトの最適化:** レイテンシやトークン消費量が問題となる場合、プロンプトの簡潔化、必要な情報の厳選、より効率的なモデルへの切り替えなどを検討します。
    *   **非同期処理:** Activityは非同期で設計されていますが、`client.models.generate_content` 呼び出し以外の部分で同期的なブロッキングI/Oがないか確認します。
    *   **Workerリソース:** WorkerのCPU、メモリ、ネットワークリソースが十分であることを確認します。リソースが不足していると、Activityの実行が遅延したり、エラーが発生したりする可能性があります。

これらの運用上の考慮事項は、本Activityを安定かつ効率的にSOP生成パイプラインの一部として機能させるために不可欠です。

from pydantic import BaseModel, Field
from typing import Optional

class LLMResult(BaseModel):
    """
    LLM からの応答と関連するメトリクスを格納するデータモデル。
    """
    text: str = Field(
        ...,
        description="LLMからの生成テキスト（例: アウトライン、SOP草稿、レビュー結果）。",
        examples=["# SOP\n## 1. はじめに\n..."],
    )
    model: str = Field(
        ...,
        description="応答を生成するために使用されたLLMモデルの識別子。",
        examples=["gemini-2.5-flash"],
    )
    input_tokens: int = Field(
        ...,
        description="LLMへの入力に消費されたトークン数。",
        ge=0, # 0以上の整数
    )
    output_tokens: int = Field(
        ...,
        description="LLMの出力に消費されたトークン数。",
        ge=0, # 0以上の整数
    )
    total_tokens: int = Field(
        ...,
        description="合計トークン数 (入力 + 出力)。",
        ge=0, # 0以上の整数
    )
    latency_ms: float = Field(
        ...,
        description="LLM API呼び出しのレイテンシ（ミリ秒）。",
        ge=0.0, # 0.0以上の浮動小数点数
    )
    error: Optional[str] = Field(
        None,
        description="エラーが発生した場合のエラーメッセージ。成功時には None。",
        examples=["API_KEY_NOT_SET"],
    )
