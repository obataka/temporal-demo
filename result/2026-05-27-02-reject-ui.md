## A. System Interaction Flow

```
ブラウザ（index.html）
  [ステータス取得後]
    → setRejectEnabled(true)   // awaiting_pr_approval の場合のみ
    → feedbackInput が活性化（白背景、入力可能）
    → rejectBtn が活性化（rose-500）

  [差し戻しボタン押下]
    → rejectWorkflow()
      → feedbackInput.value が空 → warning トースト → return
      → POST /api/reject  { workflowId, feedbackComment }
        → Hono → handle.signal("reject_with_feedback", { comment })
          → Temporal Server → Python ワークフロー
      ← 200 OK
    → 「修正指示を送信しました」トースト
    → textarea クリア、両ボタン無効化
    → statusBadge → 「差し戻し済み」
```

## C. Change Intent & Critical Points

### 設計の意図
既存の `setApproveEnabled()` パターンを踏襲し、差分を最小限に抑えた。
テキストエリアとボタンの状態を `setRejectEnabled()` で一元管理することで、
活性/非活性の同期漏れを防いでいる。

### クリティカル・ポイント（最大3点）

1. **承認ボタンのクラス変更**: `w-full` → `flex-1` に変更。横並びレイアウトのために必要だが、
   `setApproveEnabled()` 内のクラス定義も合わせて修正済み。
2. **テキストエリア空チェック**: `feedbackInput.value.trim()` が空の場合は API を呼ばず warning トーストで止める。
   バックエンド側も同様にバリデーションしているが、UX のために二重チェックしている。
3. **承認成功時の差し戻し側無効化**: `approveWorkflow()` 成功時に `setRejectEnabled(false)` と
   `feedbackInput.value = ''` を追加。承認後に差し戻しボタンが残らないよう両者を連動させている。
