## C. 設計の意図とクリティカルポイント

### 設計の意図
「意図的バグ注入 → AI 自律修正ループ → 人間承認ゲート → PR 発行」という Temporal の Durable Execution を
一本の実証スクリプトで可視化することが目的。禁止用語の注入は Phase 3 フィードバック Signal 経由で行い、
LLM が自然に生成するテキストに依存せず確実に検証シナリオを成立させる。

### クリティカルポイント 1: workflow reset による部分再実行
Run 3 が gemini-2.5-flash のクォータ枯渇で失敗した際、`temporal workflow reset --event-id 34` で
Phase 3 2回目開始直前にリセット。Phases 1-3(1回目) の結果はすべてイベント履歴から復元され、
Gemini API 呼び出しは Phase 3 2回目の **1回のみ**で完了した。
Activity コードのみの変更（モデル切り替え）であれば、workflow 全体を再実行せずに済む。

### クリティカルポイント 2: バリデーション PASS（Phase 4 修正ループ不要）
Run 2・3 では `"仮"` を禁止用語リストに含めていたため、日本語テキストの複合語（仮定・仮説）に誤マッチし
Phase 4 の 3 回修正試行を消費していた。`"仮"` を禁止用語から除外した結果、
Phase 4 は初回バリデーションで PASS となり、修正ループなしで PR 承認ゲートに到達した。

### クリティカルポイント 3: 3回の実行失敗から得た教訓
| Run | 失敗原因 | 机上デバッグで防げたか |
|---|---|---|
| Run 1 | `DUMMY_SOURCE_CODE` に禁止用語が含まれていた | ✓ grep で即検知可能 |
| Run 2 | `"仮"` の substring 検索が複合語に誤マッチ | ✓ ロジック読解で即検知可能 |
| Run 3 | Gemini free tier 上限（20 req/day）を超過 | ✓ 事前クォータ確認で防止可能 |
