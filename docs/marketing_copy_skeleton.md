### 1-B. 導入セクション：「なぜ、手順書は化石化するのか」

**セクション見出し：**

```
あなたの組織だけの問題ではない。
手順書の形骸化には、構造的な原因がある。
```

**4 つの構造的原因（箇条書きカード形式）：**

| # | 原因 | 説明 |
|---|---|---|
| 1 | **更新コストの非対称性** | 手順書を書く工数は膨大だが、書かなくても短期的には誰も困らない |
| 2 | **オーナーシップの曖昧さ** | 「誰が更新すべきか」が不文律になっており、結果として誰も動かない |
| 3 | **変化速度との乖離** | システム・法令・体制の変化スピードに、手作業での更新が追いつかない |
| 4 | **読まれない手順書の連鎖** | 古いという不信から誰も参照せず、さらに更新の優先度が下がる悪循環 |

**リスクの構造化（強調ブロック）：**

```
形骸化した手順書が引き起こす 3 つの組織リスク

⚠️  監査リスク：
    SOP 不備はコンプライアンス審査の最頻出指摘事項です。
    1 件の指摘が、開発チームへの信頼を失墜させます。

⚠️  ガバナンスリスク：
    「誰が、何を、いつ判断したか」が記録されない組織は、
    インシデント発生時に説明責任を果たせません。

⚠️  業務品質リスク：
    現場担当者が古い手順を踏む → ミス → 手戻り → 信頼コスト。
    この連鎖は、SOP が最新である限り発生しません。
```

### 1-C. ソリューションセクション：「5 フェーズの完全自律ガバナンス」

**セクション見出し：**

```
AI が、SOP 改善の全工程を自律で完走する。
```

**サブ見出し：**

```
人間は「差し戻す」か「承認する」かを判断するだけ。
その間に、AI が議論し、修正し、記録します。
```

**5 フェーズ 説明（ステップ形式）：**

```
Phase 1 ── 章立て提案（Outline）
  AI が SOP の全体構成を提案します。
  人間はその骨子を承認するか、フィードバックを送るだけです。

Phase 2 ── 詳細執筆（Draft）
  承認された構成に沿って、AI が本文を執筆します。
  専門用語・手順の具体性・コンプライアンス要件を自動的に考慮します。

Phase 3 ── 最終レビュー（Review）
  AI が自らの出力を批評し、品質を確認します。
  誤記・抜け漏れ・表現の曖昧さを自動検出します。

Phase 4 ── Writer ⇄ Reviewer 自律修正ループ（Autonomous Fix）
  Writer エージェントと Reviewer エージェントが相互に議論し、
  最良の表現に収束するまで自律的に繰り返します。
  このプロセス全体が、リアルタイムで可視化されます。

Phase 5 ── PR 自動生成・組織統合（GitHub Integration）
  承認後、変更内容がバージョン管理システムに自動統合されます。
  SOP の変更履歴が、コードと同じ厳密さで追跡・管理されます。
```

**デモ動画への誘導（インライン CTA）：**

```

```

### 1-D. 信頼性の城壁セクション：「なぜ Temporal なのか」

**セクション見出し：**

```
サーバーが落ちても、
業務プロセスは 1 ミリもバグらない。
```

**コアメッセージ（強調ブロック）：**

> Temporal は、ワークフローの全ステップを永続化するオーケストレーションエンジンです。  
> AI が Phase 3 まで進んだ状態でサーバーが障害を起こしても、  
> 復旧後に Phase 3 の正しい地点から再開します。途中経過のロストはゼロです。

**信頼性の 4 本柱：**

```
🔒  耐障害性
    ネットワーク断・サーバー停止・コンテナクラッシュ——いかなる障害にも、
    ワークフローは必ず正しい位置から再開します。

📋  完全監査ログ
    誰が、いつ、何を承認・差し戻ししたか。
    すべてのアクションが、改ざんできない履歴として自動記録されます。

🔄  冪等な再試行
    AI の呼び出しが失敗した場合、Temporal が自動でリトライします。
    同じ処理が二重に実行されないことを、エンジンレベルで保証します。

📊  リアルタイム可視化
    ワークフローの現在フェーズ・エージェントの稼働状態・ログが、
    Web UI でリアルタイムに確認できます。ブラックボックスにしません。
```

**ターゲット向けの一文：**

> 「本番環境に導入できるか」という問いへの答えは、  
> Temporal のトラックレコード（Uber / Netflix / Airbnb 等）が証明しています。

## 3. CTA 刈り取り要件

---

### 3-A. オファーの設計原則

購買検討段階の意思決定者が「ボタンを押す」ためには、以下の心理的障壁を順番に取り除く必要があります。

| 障壁 | 取り除くための設計 |
|---|---|
| 「何を渡されるかわからない」 という不安 | 提供内容を具体的に明記（60 分ヒアリング、アジェンダ事前送付） |
| 「売り込まれる」 という警戒 | 「無償」「一緒に整理する」という協力姿勢の言葉を使う |
| 「枠が余っているのでは」 という疑念 | 限定数の根拠を示す（5 社で開発者1名が対応できる上限） |
| 「大げさでは」 という懐疑 | 誇大表現を一切排除し、できることとできないことを正直に伝える |

### 3-D. 「誇大表現ゼロ」チェックリスト

本コピー全体において、以下を徹底して排除しています。

| 避けた表現 | 理由 |
|---|---|
| 「完璧な SOP が生成される」 | AI の出力品質は入力・LLM に依存するため保証不可 |
| 「導入するだけで工数がゼロになる」 | Human-in-the-Loop による人間の承認工数は残る |
| 「すべての企業に対応」 | PoC を通じて適合性を確認する必要があるため |
| 「競合他社より優れている」 | 比較根拠のない優位性主張は信頼を損なう |
