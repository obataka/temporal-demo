## 改善点リスト

SOP草稿を拝見しました。デモ動画本番撮影リハーサルという目的を明確にし、参加者の理解を助けるための構成は非常に良いと感じました。しかし、より実践的で、明確な手順書として機能させるために、いくつかの改善点があります。

### 1. 目的とスコープの明確化
*   **現状:** 「Temporal × Hono HITL 統合検証のリハーサル」というテーマですが、SOP草稿の「3.3.1. 期待される結果」や「4.1. 質疑応答」で Hono の連携について触れられているものの、**リハーサル手順自体はTemporalとダミーモジュールに終始しており、Hono がどのように関わるかの具体的なシナリオや検証項目が欠けています。**
*   **改善点:**
    *   リハーサルで**実際にHonoのAPIを呼び出す、あるいはHonoからのレスポンスを模倣するシナリオ**を明確に定義・追加してください。
    *   「2.1. リハーサルの背景と目的」において、Hono がHITLプロセスにどのように貢献するのか、そしてその連携をリハーサルでどう確認したいのかを具体的に記述してください。
    *   「3.2. リハーサル実行」のセクションに、Hono 関連の操作や確認項目を追加してください。

### 2. ダミーモジュールの役割とHITL連携の具体性
*   **現状:** ダミーモジュール (`dummy_processor.py`) の意図的な不完全性は理解できますが、これが**HITL介入の「トリガーとなる可能性」**という説明にとどまっています。リハーサルで具体的に何を「確認」するのかが不明瞭です。
*   **改善点:**
    *   `DataProcessor.validate` が `False` を返した際に、**Temporal ワークフローが具体的にどのようなアクション（例: 人間へのタスク発行、特定のイベント通知など）を行うべきか**、その期待される挙動を明確に定義してください。
    *   リハーサルでは、その「人間へのタスク発行」部分を**モック（模倣）する**のか、あるいは**実際にTemporalのAPIを叩いてタスクを発行する**のか、具体的な実行手順を記述してください。
    *   「3.2.2. HITL 要素の介入（未実装部分の確認）」で、「**この `False` を受けてワークフロー側で人へのタスク発行などの処理が入る想定です。**」とありますが、この「想定」をリハーサルでどのように「確認」するのか、具体的な確認手順（例: ワークフローのステータス確認、コンソールログの確認、Temporal UIでのタスク確認など）を追記してください。

### 3. Temporal CLIコマンドの具体性
*   **現状:** 「3.2.1. パイプライン実行」で `tctl workflow start` コマンドの例が示されていますが、`--type your_workflow_type` や `--arguments '["sample_input_1", "sample_input_2"]'` がプレースホルダーであり、**実際のリハーサルで使用する具体的なコマンドが示されていません。**
*   **改善点:**
    *   リハーサルで**実際に実行する Temporal CLI コマンド**を、ワークフロータイプ名、タスクキュー名、引数を含めて具体的に記述してください。
    *   または、Python SDK を用いてワークフローを起動するスクリプトを実行する場合、そのスクリプトの実行方法と具体的なコード（またはその一部）を示すと、より実行しやすくなります。

### 4. ログ確認の具体性
*   **現状:** 「3.2.3. ログの確認」で、確認すべきログの種類と目的は記述されていますが、**どこで（どのコンソール/ログファイル）、どのように（コマンド例など）確認するのか**が具体的に示されていません。
*   **改善点:**
    *   Temporal Worker のログ、アプリケーション（ダミーモジュール）のログなど、**出力されるログの場所を明確に**してください。
    *   ログをリアルタイムで追跡するためのコマンド例（例: `kubectl logs -f <pod_name>`、`docker logs -f <container_name>`、`tail -f <log_file>`など、環境に応じたもの）を追記してください。
    *   「3.3.1. 期待される結果」や「3.2.1. 2. `DataProcessor.process` の呼び出しと `None` 返却の確認」などのセクションでも、**「ログで確認する」という指示だけでなく、「具体的にどのようなログメッセージを確認するのか」**を明確に記述してください。

### 5. 実施体制の明確化
*   **現状:** 役割分担は定義されていますが、「担当者名 または 役職」がプレースホルダーになっています。
*   **改善点:**
    *   リハーサル実施前に、**実際の担当者名を記入**してください。
    *   可能であれば、各担当者の**具体的な責任範囲**（例: 「実行者は、ファシリテーターの指示に従い、指定されたコマンドを正確に実行する責任を負う」など）を追記すると、よりスムーズな進行に繋がります。

### 6. Hono連携シナリオの追加（最重要）
*   **現状:** SOP草稿全体を通して、Hono の連携がリハーサル手順に具体的に落とし込まれていません。
*   **改善点:**
    *   **シナリオ例:**
        *   Hono からのデバイスデータ受信をトリガーに Temporal ワークフローが開始される。
        *   Temporal ワークフローが Hono API を呼び出し、特定のデータを取得する（またはHono経由で外部DBに保存する）。
        *   HITL 介入が必要と判断された後、その結果が Hono 経由でデバイスに通知される。
    *   これらのシナリオを「2.1. リハーサルの背景と目的」や「3.1.3. シナリオ確認」に追加し、「3.2. リハーサル実行」にも具体的な手順を盛り込んでください。Hono のモックサーバーを用意する、あるいは Hono が実際に起動している環境でリハーサルを行うなど、実施方法も明確にしてください。

### 7. 用語の定義の補足
*   **現状:** 用語の定義は概ね適切ですが、リハーサルとの関連性が薄いものもあります。
*   **改善点:**
    *   「ダミーモジュール」の定義を、リハーサルで**意図的に不完全な状態にしておくことで何を検証したいのか**という文脈で補足すると、より理解しやすくなります。
    *   「HITL」の定義で、「未実装のバリデーション処理が HITL による介入のトリガーとなります」とありますが、**リハーサルで「未実装」をどう確認し、それがHITL介入にどう繋がるはずなのか**を、より具体的に説明する必要があります。

### 8. SOPのバージョン管理・承認欄
*   **現状:** SOPとして標準的な管理情報がありません。
*   **改善点:**
    *   SOPの冒頭に、**バージョン番号、作成日、改訂日、作成者、承認者**などの欄を追加してください。

### 1.4. 用語の定義

*   **技術的な動作確認:** Temporal と Hono が連携し、HITL 要素が期待通りに機能するかを確認します。特に、Hono 経由で受信したデータが Temporal ワークフローをトリガーし、その後の処理で HITL 介入が必要となるシナリオの連携を確認します。

#### 3.1.2. コード概要と意図的な不完全性

リハーサルで使用する主要な Python コードについて、その概要と意図的な不完全性を理解します。

*   **`data_processor.py` (アクティビティ内から呼び出されるモジュール):**
    *   **クラス:** `DataProcessor`
        *   **`process(self, data: dict) -> dict | None` メソッド:**
            *   **概要:** 入力された辞書形式のデータを処理するメソッドです。
            *   **リハーサルでの意図的な不完全性:** このメソッドは、常に `None` を返却するように実装されています。これは、「データ処理は試行されたが、有効な結果が得られなかった」という状況をシミュレートします。
            *   **ログ:** `logger.debug("Processing data: %s", data)` により、メソッドが呼び出された際の入力データがデバッグログとして出力されます。
        *   **`validate(self, item: object) -> bool` メソッド:**
            *   **概要:** アイテムの妥当性を検証するメソッドです。HITL による介入のトリガーとなることを想定しています。
            *   **リハーサルでの意図的な不完全性:** このメソッドは、**実装されていません**。そのため、`raise NotImplementedError` を発生させる、または常に `False` を返却するように実装されています。これは、「**自動的なバリデーションロジックが存在しない**」状態を示し、HITL による介入が必須となる状況を表現します。
*   **`hitl_activity.py`:**
    *   `process_data_activity(input_data: dict)`: `DataProcessor.process` を呼び出すアクティビティ。`None` が返却されることを確認します。
    *   `validate_data_activity(item: object)`: `DataProcessor.validate` を呼び出すアクティビティ。`NotImplementedError` または `False` の返却を確認します。
*   **`hitl_workflow.py`:**
    *   `HitlWorkflow`: Temporal ワークフロー定義。
        *   Hono からのデータ受信をトリガーとする (リハーサルでは手動開始)。
        *   `process_data_activity` を呼び出す。
        *   `process_data_activity` の結果が `None` の場合、「処理失敗」としてログを出力する。
        *   本来であれば、`validate_data_activity` を呼び出し、その結果 (`False` または `NotImplementedError`) をもって、**人間へのタスク発行 (ActivityTask)** をトリガーする設計を想定。
            *   **リハーサルでの確認:** `validate_data_activity` が呼び出されたこと、および `NotImplementedError` が発生するか `False` が返却されることを確認します。また、**この `False`/`NotImplementedError` を受けて、ワークフローが「人による確認が必要」な状態に遷移するはずである**という点を認識します。
            *   **注意:** 本リハーサルでは、**実際に人へのタスク発行機能までを実装・検証するのではなく**、HITL 介入の**トリガーとなる「未実装/失敗」の状態を確認すること**に主眼を置きます。
*   **`send_to_hono.py` (補助スクリプト):**
    *   Hono API を使用して、ダミーデータを Hono に送信するスクリプト。リハーサルのトリガーとして使用します。

#### 3.1.3. シナリオ確認

本リハーサルでは、以下の HITL 統合検証シナリオを実施します。

**シナリオ:** Hono 経由で受信したデータに対する処理と、未実装のバリデーションによる HITL 介入トリガーの確認

1.  **データ送信:** `send_to_hono.py` スクリプトを使用し、ダミーデータを Hono へ送信します。
2.  **ワークフロー開始:** Hono でデータ受信がトリガーとなり、Temporal ワークフロー (`HitlWorkflow`) が開始される（リハーサルでは手動開始または Hono インテグレーションを模倣）。
3.  **データ処理:** ワークフロー内で `process_data_activity` が呼び出され、`DataProcessor.process` が実行されます。
4.  **処理結果確認（失敗）:** `DataProcessor.process` が `None` を返却します。ワークフローはこれを「処理失敗」と判断し、警告ログを出力します。
5.  **バリデーション実行:** ワークフローは次に `validate_data_activity` を呼び出します。
6.  **HITL トリガー確認:** `DataProcessor.validate` が未実装であるため、`NotImplementedError` が発生するか、`False` が返却されることを確認します。
7.  **HITL 介入の認識:** この `False` または `NotImplementedError` が、本来であれば「人による確認が必要」という HITL 介入のトリガーとなることを認識します。
8.  **ログの確認:** 各ステップで期待されるログ（デバッグログ、警告ログ、エラーログ）が出力されていることを確認します。

#### 3.1.4. Temporal Worker の起動

### 3.2. リハーサル実行

以下の手順に従って、リハーサルを実行します。

#### 3.2.1. Hono へのデータ送信とワークフロー開始

1.  **Hono へのデータ送信:**

    ```bash
    # 例: サンプルデータ (JSON形式) を Hono の /devices/{deviceId}/messages エンドポイントへ POST
    curl -X POST -H "Content-Type: application/json" -d '{"timestamp": 1678886400, "sensorId": "sensor-123", "value": 42.5}' http://localhost:8080/devices/device-abc/messages
    ```
    *   **確認:** Hono サーバーまたは Hono Device Simulator のログで、データが正常に受信されたことを確認します。

2.  **Temporal ワークフローの開始:**
    *   Hono でのデータ受信が Temporal ワークフローをトリガーするシナリオを模倣するため、手動でワークフローを開始します。Temporal CLI を使用します。

    ```bash
    # 例: ワークフローIDはユニークに、タスクキュー名とワークフロータイプは Worker で定義したものに合わせる
    tctl workflow start --workflow_id rehearsal_hitl_$(date +%s) --task_queue rehearsal_hitl_task_queue --type HitlWorkflow --arguments '{"hono_data": {"timestamp": 1678886400, "sensorId": "sensor-123", "value": 42.5}}'
    ```
    *   *注:* `--arguments` には、ワークフローの `__init__` メソッドに渡される引数を JSON 文字列として指定します。実際の引数名は `hitl_workflow.py` の定義に合わせてください。
    *   **確認:** Temporal CLI の出力で、ワークフローが正常に開始されたことを確認します。

#### 3.2.2. Temporal Worker による処理実行とログ確認

1.  **`process_data_activity` の実行と `None` 返却の確認:**
    *   Worker がワークフローからのリクエストを受け取り、`process_data_activity` を実行します。

        *   **Worker ログ:** `hitl_workflow.py` のロジックにより、「処理失敗」を示す警告ログ（例: `logger.warning("Data processing returned None. Skipping.")`）が出力されていることを確認します。

2.  **`validate_data_activity` の実行と HITL トリガー確認:**
    *   ワークフローは次に `validate_data_activity` を呼び出します。
    *   **期待される挙動:** `DataProcessor.validate` メソッドが未実装のため、`NotImplementedError` が発生するか、`False` が返却されます。
    *   **確認方法:**
        *   **Worker ログ:** `data_processor.py` の `DataProcessor.validate` メソッドが呼び出された（または呼び出しを試みた）ことを示すログ、または `NotImplementedError` のエラーログが出力されていることを確認します。
        *   **ワークフローの状態:** `tctl workflow describe --workflow_id <workflow_id>` コマンドなどでワークフローの状態を確認し、エラーが発生しているか、あるいは意図した状態（例: 異常終了、または手動介入待ちへ遷移するはずだった状態）になっていることを確認します。
        *   **HITL 介入の認識:** この `False` または `NotImplementedError` が、本来であれば「人による確認が必要」という HITL 介入のトリガーとなることを、ファシリテーターの説明のもと、参加者全員で認識します。

リハーサル終了後、観測者および記録者は以下の観点から結果を評価します。

#### 3.3.1. 期待される結果

リハーサルが成功した場合、以下の挙動が確認されるはずです。

#### 3.3.2. 問題点の特定

リハーサル中に発見された問題点を、以下のカテゴリに分けて詳細に記録します。

*   **コードの挙動に関する問題:**
    *   予期しないエラーが発生した（例: `NullPointerException`、`KeyError` など）。
    *   ログ出力が不足している、または間違っている。
    *   メソッドの返り値が期待通りでない。
    *   Temporal Worker がクラッシュした。
*   **環境に関する問題:**
    *   Temporal クラスタ/Worker が正しく起動しない。
    *   ライブラリの依存関係の問題。
    *   Hono への接続/通信の問題。
    *   ログの取得が困難。
*   **手順に関する問題:**
    *   リハーサル手順が不明確で、実行に迷った。
    *   コマンドの誤りや、実行順序の間違い。
    *   HITL 介入のトリガーとなるべき箇所が不明瞭。
    *   Hono と Temporal の連携フローが理解しにくい。

**記録方法:** 問題が発生した具体的なステップ、エラーメッセージ、観測された挙動、その影響、および発見日時を詳細に記録してください。
