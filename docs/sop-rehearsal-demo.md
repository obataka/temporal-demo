## デモ動画本番撮影リハーサル用 SOP（Temporal × Hono HITL 統合検証）

## 1. はじめに

### 1.1. 本 SOP の目的
本 SOP は、Temporal と Hono を用いた Human-in-the-Loop (HITL) 統合検証のデモ動画本番撮影リハーサルを、**効率的かつ効果的に実施するための標準手順を定めるものです。**本リハーサルを通じて、本番撮影での手戻りを最小限に抑え、システム連携の潜在的な問題を早期に発見し修正コストを低減することで、デモ動画の品質向上とプロジェクトの効率化を図ります。具体的には、以下の達成を目指します。

*   **システム連携の安定性確認:** Temporal、Hono、および連携する Python コード間のデータ連携が意図した通りに行われることを確認します。
*   **各コンポーネントの動作確認:** Temporal ワークフロー、Hono サーバー、および Python ダミーモジュールの各機能が正常に動作することを確認します。
*   **HITL 要素の組み込み確認:** Human-in-the-Loop の仕組み（人間による承認プロセス、例外処理、データ検証など）がシステムに正しく組み込まれ、期待通りに機能することを確認します。
*   **問題発生時の対応フロー検証:** 予期せぬ問題発生時に、迅速かつ正確なログ収集、原因特定、およびエスカレーションが行われることを確認します。
*   **本番撮影に向けた課題の特定と改善:** リハーサルを通じて明らかになったシステム上の課題、手順上の不備、またはドキュメントの不足点を特定し、本番撮影前の改善に繋げます。

### 1.2. 適用範囲
本 SOP は、Temporal × Hono HITL 統合検証のデモ動画本番撮影リハーサルに関わる全ての関係者（開発者、テスター、オペレーター、マネージャー等）および、リハーサルに関連する全ての準備、実施、評価作業に適用されます。

### 1.3. 用語の定義
*   **Temporal:** 分散システムにおけるワークフローオーケストレーションプラットフォームを指します。本 SOP では、ビジネスロジックの実行順序と状態管理を担います。
*   **Hono:** IoT デバイスからのデータ収集、処理、および連携を行うためのオープンソースプラットフォームを指します。本 SOP では、データの前処理と後処理、および外部システムとの連携ゲートウェイとして機能します。
*   **HITL (Human-in-the-Loop):** 人間がシステムの一部として意思決定や介入を行うプロセスを指します。本 SOP では、Temporal ワークフロー内で人間の承認や確認を必要とするステップを検証します。
*   **パイプライン:** データがシステム内を流れる一連の処理ステップを指します。本 SOP では、データ収集 → Hono での前処理 → Temporal でのオーケストレーション → 人間による確認（HITL） → Hono での後処理 → 結果記録、といった具体的な流れを想定します。
*   **ダミーモジュール:** 本番環境のモジュールを模倣しつつ、意図的に不完全な実装を含む Python コードを指します。本リハーサルでは、エラーハンドリングのテスト、ロギングの確認、システム全体の堅牢性テストなどを検証するために使用されます。

## 2. 環境構築

### 2.1. 役割と責任
本リハーサルにおける主要な役割と責任は以下の通りです。

*   **パイプライン実行者 (Python):**
    *   Python スクリプトの実行と初期データの投入を行うこと。
    *   Python コードの動作状況の確認を行うこと。
    *   問題発生時の初期切り分けと情報収集を行うこと。
*   **観測者 (Temporal):**
    *   Temporal UIでのワークフロー状態遷移、アクティビティの実行状況、エラーログの確認を行うこと。
    *   ワークフローの期待される動作との比較検証を行うこと。
    *   Temporal 関連の問題の特定と報告を行うこと。
    *   特に、アクティビティのリトライ動作やタイムアウト設定が意図通りに機能しているかを確認すること。
*   **観測者 (Hono):**
    *   Honoのアクセスログにおけるリクエスト/レスポンスコード、カスタムログ、エラーログの確認を行うこと。
    *   Hono サービス間のデータ連携の確認を行うこと。
    *   Hono 関連の問題の特定と報告を行うこと。
*   **観測者 (Python):**
    *   Pythonスクリプトの標準出力、エラー出力、アプリケーションログの確認を行うこと。
    *   ダミーモジュールの各関数の実行状況と出力の確認を行うこと。
    *   Python コード関連の問題の特定と報告を行うこと。
*   **エスカレーション担当:**
    *   問題発生時、Slackチャンネルへの投稿、Jira/Backlog等のチケット起票、または担当者への直接連絡（緊急時）により、速やかに担当者や関係部署に連絡し、対応を依頼すること。
    *   問題解決に向けた関係者間の調整を行うこと。

### 2.2. 実施環境の準備
リハーサルを実施するための環境を以下の手順で準備します。

1.  **Temporal サーバーの起動:**
    *   Temporal サーバーと UI を Docker Compose で起動します。
    ```bash
    docker-compose -f docker-compose-temporal.yml up -d
    ```
    *   **成功基準:** Temporal UIが `http://localhost:8080` でアクセス可能であり、エラーが表示されていないことを確認します。
2.  **Hono サーバーの起動:**
    *   Hono サーバーを Docker Compose で起動します。
    ```bash
    docker-compose -f docker-compose-hono.yml up -d
    ```
    *   **成功基準:** Dockerコンテナが正常に起動し、Honoのログにエラーが出力されていないことを確認します。Honoのエンドポイントが期待通りに動作することを確認します（例: `curl http://localhost:8080/health`）。
3.  **Python 実行環境の準備:**
    *   必要な Python ライブラリをインストールします。
    ```bash
    pip install temporalio hono-client requests
    ```
    *   **成功基準:** 全てのライブラリがエラーなくインストールされたことを確認します。
4.  **ダミーモジュールの配置:**
    *   `dummy_module.py` および `pipeline_workflow.py` を適切なディレクトリに配置します。
    *   **成功基準:** 配置されたファイルがPython実行環境からインポート可能であることを確認します。

## 3. リハーサル実施手順

### 3.1. リハーサル開始前の確認
*   全ての関係者が役割と責任を理解していることを確認すること。
*   各観測者がそれぞれの監視ツール（Temporal UI、Hono ログ、Python コンソールなど）を開き、準備ができていることを確認すること。
*   ネットワーク接続が正常であることを確認すること。
*   Temporal Workerが起動していることを確認すること。

### 3.2. パイプラインのトリガーと正常系検証

#### 3.2.1. `run_pipeline` 関数への入力データ（`inputs`）の指定方法
`pipeline_workflow.py` 内の `run_pipeline` 関数に渡す入力データを定義します。リハーサルでは、正常系データ、異常系データ（例: 必須項目欠落、不正なデータ形式）、および境界値データを組み合わせて検証します。

```python
# クライアントコード (例: client.py) からワークフローを起動する際の入力データ例
# 正常系データ
inputs_normal = {
    "device_id": "device-001",
    "timestamp": "2023-10-27T10:00:00Z",
    "temperature": 25.5,
    "humidity": 60.2,
    "status": "active"
}
```

#### 3.2.2. パイプラインの実行
Pythonクライアントスクリプトを実行し、Temporalワークフローをトリガーします。

```bash
# 例: Pythonクライアントスクリプト (client.py) を実行
python client.py --input-data '{"device_id": "device-001", "timestamp": "2023-10-27T10:00:00Z", "temperature": 25.5, "humidity": 60.2, "status": "active"}'
```
*   **成功基準:** Pythonスクリプトがエラーなく完了し、Temporal UIで新しいワークフローが「Running」または「Completed」ステータスで表示されることを確認します。

#### 3.2.3. 正常系動作の確認
パイプライン実行後、各コンポーネントが期待通りに動作したことを確認します。

*   **観測者 (Temporal):**
    *   Temporal UIでワークフローが正常に完了（`Completed`）していることを確認します。
    *   ワークフローのHistoryで全てのアクティビティが`Completed`になっていることを確認します。
    *   HITLステップで人間による承認が適切に行われたことを確認します。
*   **観測者 (Hono):**
    *   Honoのアクセスログに、Temporalからのリクエストが正常に処理され、`200 OK`などの成功レスポンスが記録されていることを確認します。
    *   Honoが外部システムへデータを連携した場合、その連携が成功していることを確認します。
*   **観測者 (Python):**
    *   Pythonスクリプトの標準出力に、最終結果が期待通りに出力されていることを確認します。
    *   ダミーモジュールのログに、各関数が正常に実行されたことを示すメッセージが記録されていることを確認します。

### 3.3. 異常系シナリオの実行と検証

#### 3.3.1. 異常系入力データの準備
セクション3.2.1で定義した異常系データ（例: 必須項目欠落、不正なデータ形式）を用いて、パイプラインをトリガーします。

```python
# 例: 異常系データ (必須項目欠落)
inputs_missing_field = {
    "device_id": "device-002",
    "timestamp": "2023-10-27T10:05:00Z",
    "temperature": 26.0,
    # "humidity" が欠落
    "status": "inactive"
}
```

#### 3.3.2. Temporalの自動リトライ検証 (503エラー時)
Honoが一時的に503エラーを返す状況を想定し、Temporalのアクティビティ自動リトライ動作を検証します。

1.  **前提:** Honoが連携する外部サービスが一時的に不安定になり、503エラーを返す可能性がある状況を想定します。または、テスト用にHonoが特定の条件で503を返すように設定されていることを確認します。
2.  **実行:** パイプラインをトリガーします。
    ```bash
    # 例: Pythonクライアントスクリプトを実行
    python client.py --input-data '{"device_id": "device-005", "timestamp": "2023-10-27T10:15:00Z", "temperature": 20.0, "humidity": 55.0, "status": "active"}'
    ```
3.  **観測 (Temporal UI):**
    *   Temporal UIで該当ワークフローの「History」タブを開きます。
    *   Hono呼び出しに関連するActivityが「Failed」ステータスになった後、すぐに「Scheduled」ステータスで再スケジュールされることを確認します。
    *   Activityの「Attempts」カウントが増加することを確認します。
    *   リトライ間隔（バックオフ）が適用され、次の実行までの間隔が徐々に長くなることを確認します。
    *   **成功基準:**
        *   一時的な問題が解消された場合、Activityが最終的に「Completed」になり、ワークフローが正常に続行されることを確認します。
        *   最大リトライ回数を超えた場合、Activityが最終的に「Failed」となり、ワークフローがエラーパスに進む（または設定されたフォールバック処理が実行される）ことを確認します。
4.  **観測 (Honoログ):**
    *   Honoのログで503エラーが記録されていることを確認します。
    *   Temporalからのリトライリクエストが複数回Honoに到達していることを確認します。
    *   **成功基準:** 503エラーが一時的であった場合、最終的にHonoが200 OKを返し、処理が成功していることを確認します。
5.  **観測 (Python Workerログ):**
    *   Temporal Workerのログで、Activityの実行失敗とリトライに関するログが出力されていることを確認します。
    *   **成功基準:** リトライポリシーに従って、Activityが再実行されていることを示すログが確認できること。

#### 3.3.3. その他の異常系動作の確認
*   **観測者 (Temporal):**
    *   ワークフローが`Failed`ステータスになること、または定義されたエラーハンドリングパスに進むことを確認します。
    *   特定のActivityが`Failed`または`Timed Out`になっていることを確認します。
*   **観測者 (Hono):**
    *   Honoのアクセスログに、入力データに応じたエラーコード（例: `400 Bad Request`）が記録されていることを確認します。
    *   HonoがエラーレスポンスをTemporalに正しく返していることを確認します。
*   **観測者 (Python):**
    *   Pythonスクリプトの標準出力またはログに、期待されるエラーメッセージが出力されていることを確認します。
    *   ダミーモジュールがエラーハンドリングロジックを正しく実行していることを確認します。

### 3.4. 問題発生時の対応フロー検証
リハーサル中に予期せぬ問題が発生した場合、以下の手順で対応します。

1.  **問題の特定:** 各観測者は自身の担当範囲で問題（エラーログ、予期せぬ動作など）を特定します。
2.  **情報収集:** 問題発生時の状況（タイムスタンプ、入力データ、エラーメッセージ、関連ログ）を詳細に記録します。
3.  **初期切り分け:** どのコンポーネントで問題が発生している可能性が高いか、初期的な切り分けを行います。
4.  **エスカレーション:** エスカレーション担当は、収集した情報を基に速やかにSlackチャンネルへの投稿、Jira/Backlog等のチケット起票、または担当者への直接連絡を行います。
5.  **共同での原因特定と修正:** 関係者間で協力し、問題の原因を特定し、可能な場合はその場で修正を試みます。

### 3.5. リハーサル終了後の確認とフィードバック
*   全てのリハーサルシナリオが実行されたことを確認すること。
*   発見された課題、問題点、改善点をリストアップすること。
*   本番撮影に向けた改善計画を策定すること。
*   本 SOP の内容に不備がないか確認し、必要に応じて改訂すること。
