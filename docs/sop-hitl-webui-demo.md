# Temporal × Hono HITL 統合検証 SOP

## 1. はじめに

### 1.1. ドキュメントの目的

本SOPは、Temporal WorkflowとHono Human-in-the-Loop (HITL) 機能を統合した検証の初期段階における、Pythonベースのダミー実装の実行手順と動作検証方法を記述します。これにより、開発者はこの統合シナリオの基本的な概念を理解し、今後の開発に向けた基盤を確立することを目的とします。

### 1.2. 対象読者

本SOPは、以下のスキルセットを持つ読者を対象としています。

*   Pythonプログラミングの基礎知識
*   Temporal Workflowの基本的な理解
*   DockerおよびDocker Composeの操作経験
*   Web UIアプリケーションの基本的な理解

### 1.3. Temporal × Hono HITL 統合検証の概要

本検証は、Temporal Workflowが人間の承認を必要とするタスクをHonoに発行し、HonoのWeb UIを通じて人間がそのタスクを承認または却下する、という一連のプロセスをシミュレートするものです。

**連携フローのイメージ:**

```mermaid
graph TD
    A[Temporal Workflow] --> B{データ処理タスクの実行};
    B --> C{人間の承認が必要か？};
    C -- はい --> D[Honoへのタスク発行];
    D --> E[Hono Web UI];
    E --> F[ユーザー操作 (承認/却下)];
    F --> G[HonoからTemporalへの結果通知];
    G --> H{ワークフロー続行/分岐};
    C -- いいえ --> H;
```

本SOPで扱うソースコードは、この連携の基礎となるPython部分の**初期段階のダミー実装**です。具体的には、`DataProcessor` クラスの`process`および`validate`メソッドが現在ダミーであり、`run_pipeline` 関数でそのダミー実装がどのようにワークフローに影響するかを検証します。

## 3. システムアーキテクチャと連携フロー

### 3.1. 連携する Temporal と Hono の役割と連携フロー

本統合検証におけるTemporalとHonoの主要な役割と連携フローは以下の通りです。

1.  **Temporal:** データ処理ワークフロー全体をオーケストレーションします。人間の介入（HITL）が必要なポイントでHonoへタスクを発行します。
2.  **Hono:** Temporalから発行された人間の介入タスク（例: 承認要求）を受け取り、Web UIを通じてユーザーに提示します。ユーザーの操作結果（承認/却下）をTemporalに通知します。

**連携フローの詳細:**

1.  **データ入力:** Temporal Workflowが、処理対象のデータを受け取ります。
2.  **データ処理 (Temporal Activity):** Workflow内で定義されたActivityが、初期データ処理を実行します。
3.  **Honoへのタスク発行 (Temporal Activity):** データ処理の結果、人間の承認が必要と判断された場合、Temporal ActivityはHonoに対して承認タスクを発行します。この際、必要なデータやコンテキストをHonoに渡します。
4.  **Honoでのタスク受付とWeb UI表示:** HonoはTemporalからのタスクを受け取り、その内容をWeb UI上に表示します。
5.  **ユーザーによる操作:** Hono Web UIにアクセスしたユーザーは、表示されたタスクを確認し、承認、却下、あるいは情報の修正などの操作を行います。
6.  **HonoからTemporalへの結果通知:** ユーザー操作が完了すると、Honoはその結果（例: 承認済み、却下済み、修正済みデータなど）をTemporal Workflowに通知します（例: SignalやActivity Callbackを使用）。
7.  **Temporal Workflowの再開/続行:** Temporal WorkflowはHonoからの通知を受け取り、その結果に基づいて後続の処理（例: データ更新、別の処理ステップへ移行、エラー処理など）を続行します。

### 3.2. Web UI 画面承認デモの全体像

本デモは、前述の連携フローのうち、Pythonダミー実装を通じて現時点でのデータ処理と、将来的なHono連携のイメージを理解することを目的としています。

**デモのフェーズ:**

*   **現在のダミーコード実行フェーズ:**
    Pythonスクリプトが擬似的なデータ処理パイプラインを実行します。`DataProcessor` クラスの`process`メソッドと`validate`メソッドはダミー実装であり、特定の条件下で無効な入力をスキップするように動作します。
*   **将来的な機能追加フェーズ（概念説明）:**
    Hono連携が完了すると、`DataProcessor`の`process`メソッドで生成されたデータは、Honoに送られ、Web UI上に承認待ちタスクとして表示されます。ユーザーはWeb UIを通じてそのデータを承認・却下し、その結果がTemporalにフィードバックされ、ワークフローが続行します。

```mermaid
graph LR
    A[Python スクリプト] --> B{入力リスト};
    B --> C[run_pipeline関数];
    C --> D[DataProcessor.process (ダミー)];
    D --> E{結果は有効か？};
    E -- Yes (将来的にHonoへ) --> F[Hono Web UI];
    F --> G[ユーザー操作];
    G --> H[HonoからTemporalへ結果通知];
    E -- No (スキップ) --> I[ログ出力];
    H --> J[ワークフロー続行/終了];
    I --> J;
```

## 4. `DataProcessor` クラス

`DataProcessor` クラスは、デモにおけるデータ処理の中核を担うプレースホルダーです。

### 4.1. クラスの概要

`DataProcessor` クラスは、入力データを受け取り、それを処理し、必要に応じて検証する機能を持つことを想定しています。現時点では、そのメソッドは将来的な実装の骨組みを示しており、具体的な処理ロジックはダミーです。

### 4.2. `process` メソッド（現時点での不完全な実装）

*   **実装意図:** 入力データ `data` を受け取り、何らかの処理を施して結果を返すことを想定しています。
*   **現在のダミー実装:** 常に `None` を返します。
*   **現在の動作:** `run_pipeline` 関数において、このメソッドが `None` を返すことで、当該入力データが無効としてスキップされることをシミュレートします。
*   **（将来的な完全な実装では）:** 実際のビジネスロジックに基づいてデータを変換、加工、またはエンリッチメントし、処理済みのデータオブジェクトを返します。この処理結果がHonoに送られる承認対象データとなることが期待されます。

### 4.3. `validate` メソッド（現時点での不完全な実装）

*   **実装意図:** 処理済みのデータが特定のバリデーションルールを満たしているかを確認することを想定しています。
*   **現在のダミー実装:** 常に `False` を返します。
*   **現在の動作:** `run_pipeline` 関数において、このメソッドが `False` を返すことで、処理済みのデータであっても後続の処理に進めないことをシミュレートします。
